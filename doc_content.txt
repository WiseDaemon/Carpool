[8] 1. Executive Summary
[9] Reliance Carpool Platform is an internal, enterprise-grade carpooling solution developed exclusively for employees of Reliance Industries Limited and its subsidiaries — including Jio Platforms, Reliance Retail, and RIL Oil-to-Chemicals (O2C). The platform aims to reduce the commuting cost burden on employees, alleviate vehicular congestion around Reliance campuses, and contribute measurably to the organisation's Environmental, Social, and Governance (ESG) commitments by reducing carbon emissions.
[10] The system is a full-stack web application built on a React (Vite) frontend with a Node.js/Express backend and an SQLite database. It supports three distinct roles — Passenger, Pool Host, and Admin — with real-time WebSocket-based notifications and a comprehensive Admin control panel.
[11] Key Highlights
[12] Role-based access control with three tiers: Passenger, Pool Host, and Admin.
[13] Live, map-assisted ride discovery using geospatial matching (Haversine algorithm, ≤3 km tolerance).
[14] Real-time push notifications via Socket.io for ride events and SOS alerts.
[15] AI-powered chatbot (Gemini 2.5 Flash) for in-app assistance and ride booking.
[16] Women-Only Rides filter for enhanced commuter safety.
[17] SOS/Incident reporting system with 15-minute SLA and immediate Admin alerts.
[18] ESG Impact Dashboard tracking CO₂ avoided, fuel saved, and employee cost savings.
[19] Secure JWT-based authentication with sliding session management and bcrypt password hashing.
[20] Enterprise-grade API security: rate limiting, CORS whitelisting, input validation, and IDOR prevention.

[22] 2. Intent & Business Objective
[23] 2.1 Problem Statement
[24] A significant portion of Reliance employees commute daily across long distances using personal vehicles. This leads to:
[25] High individual commuting costs, particularly with rising fuel prices.
[26] Traffic congestion around Reliance campuses and facilities.
[27] Increased per-capita carbon footprint, adversely affecting ESG goals.
[28] Absence of a verified, trusted carpooling platform among colleagues.
[29] 2.2 Solution
[30] The Reliance Carpool Platform provides a closed, internally-governed carpooling network where:
[31] Employees can offer or request rides along verified daily commute routes.
[32] Pool Hosts (drivers) can monetise their commute by sharing fuel costs.
[33] Passengers save money by splitting travel costs.
[34] Admins can monitor the entire platform, resolve disputes, and track ESG impact in real time.
[35] 2.3 Strategic Alignment
[36] This platform directly supports Reliance's stated ESG targets under the New Energy and Green Initiatives umbrella, contributing to Scope 3 emission reductions from employee commuting — a category reportable to frameworks such as GRI and CDP.

[44] 4. User Roles & Permissions
[45] The platform enforces a three-tier Role-Based Access Control (RBAC) model. Each role has distinct capabilities and restricted access enforced at both the frontend and backend levels.
[46] 
[47] 4.1 Passenger
[48] A Passenger is any verified Reliance employee who wishes to avail carpooling services as a rider.
[49] 
[50] 4.2 Pool Host
[51] A Pool Host is a verified Reliance employee who owns a vehicle and wishes to offer rides to colleagues.
[52] 
[53] 4.3 Admin
[54] An Admin is a designated Reliance HR or Commute Management staff member with full platform oversight. Admin accounts are seeded directly in the database; they cannot be self-registered.
[55] 
[56] 5. Core Features
[57] 5.1 Ride Discovery & Geospatial Matching
[58] The ride discovery system is the core of the Passenger experience. Passengers enter a pickup location and dropoff location either via the autocomplete search bar (powered by Photon/Komoot) or by clicking directly on an interactive Leaflet map.
[59] The backend applies the Haversine formula to calculate the great-circle distance between the passenger's requested coordinates and each listed ride's origin/destination. A ride is considered a match only if:
[60] The distance from the passenger's pickup to the ride's origin is ≤ 3 km, AND
[61] The distance from the passenger's dropoff to the ride's destination is ≤ 3 km.
[62] Matched rides are returned sorted by total deviation distance (shortest first), providing the most convenient options at the top.
[63] 
[64] 5.2 Real-Time WebSocket Notifications (Socket.io)
[65] The platform uses Socket.io to deliver instant, server-pushed notifications to active browser sessions without requiring page refreshes.
[66] Each user is placed in an isolated private room (user_{id}) on login, ensuring notifications are never broadcast to unintended recipients. Admins join a dedicated admin_room.
[67] 
[68] 5.3 AI-Powered Chatbot (Gemini Assistant)
[69] An integrated AI assistant powered by Google Gemini 2.5 Flash Lite is available within the dashboard for all logged-in users. The chatbot can:
[70] Answer questions about carpool policies and guidelines.
[71] Provide live stats and KPI summaries to users in natural language.
[72] Draft ride booking requests (using structured tool calls) that the user can confirm.
[73] List new commutes on behalf of Pool Hosts.
[74] The chatbot is grounded with a knowledge base loaded from a Reliance Carpool requirements document, enabling contextually accurate responses.
[75] 
[76] 5.4 In-App Messaging (Ride Chat)
[77] Once a ride request is accepted, both the Pool Host and Passenger can exchange messages through a dedicated, request-scoped chat interface. Messages are stored in the database against the ride_request ID and are accessible to both parties throughout the lifecycle of the request.
[78] 
[79] 5.5 Rating & Feedback System
[80] After a trip is completed, Passengers may submit a structured rating for their Pool Host. The rating system includes:
[81] A 1–5 star numerical rating.
[82] Predefined tag selection (e.g., "Punctual", "Safe Driver", "Harassment", "Rash Driving").
[83] An optional free-text comment field.
[84] Auto-Escalation Policy: If a submitted rating contains safety-related tags such as "Harassment", "Unsafe Driving", or "Rash Driving", the system automatically creates an incident ticket in the Admin's Incident Queue, flagging the driver for review without requiring a manual report.
[85] 
[86] 5.6 SOS & Incident Reporting
[87] Any logged-in user (Passenger or Pool Host) can trigger an SOS alert from within their dashboard. The report includes:
[88] A category selection (e.g., Safety Concern, Medical Emergency, Vehicle Breakdown).
[89] Automatic 15-minute SLA deadline attached to the incident.
[90] Immediate real-time notification pushed to all online Admin sessions via WebSocket.
[91] Admins can view all open incidents in the Live Incidents Queue, with SLA countdown status, and mark them as Resolved.
[92] 
[93] 5.7 ESG Impact Dashboard
[94] The Admin-facing ESG Impact Dashboard provides a visual summary of the environmental and financial benefits of the carpooling platform, including:
[95] Total CO₂ Avoided (kg) — estimated based on rides taken and ARAI fuel norms.
[96] Total Fuel Saved (litres) — aggregated across all completed rides.
[97] Employee Cost Savings (₹) — total amount saved by employees through shared rides.
[98] Active Carpool Users — total number of verified platform participants.
[99] Monthly trend charts (CO₂ savings and fuel savings) rendered using Recharts.
[100] Business Unit breakdown — showing contribution per Reliance subsidiary.
[101] 
[102] 5.8 Women-Only Rides
[103] Pool Hosts may flag a ride listing as Women-Only when creating a ride. Female Passengers can apply a filter in the ride discovery flow to exclusively see these rides. This feature is designed to improve the safety and comfort of female employees using the platform.

[105] 6. Platform Policies
[106] 6.1 Registration & Eligibility
[107] Only Reliance Industries Limited employees and affiliates are eligible to register.
[108] Corporate email addresses are required for registration and must follow a valid email format.
[109] Passwords must be at least 8 characters long.
[110] The "Admin" role cannot be self-registered; Admin accounts are created only by the system or by an existing Admin.
[111] Pool Hosts must provide vehicle details at the time of registration and must have their Vehicle Pass verified by an Admin before they can list rides.
[112] A single email address may hold both a Passenger and a Pool Host profile simultaneously (dual-role registration is permitted).
[113] 
[114] 6.2 Ride Listing Policy
[115] Pool Hosts may only list rides for genuine, planned commute routes.
[116] A Pool Host cannot list a ride if they do not have an approved Vehicle Pass (pending verification blocks ride listing).
[117] Ride prices are set by the Pool Host and must reflect fair cost-sharing, not commercial profit.
[118] The number of seats offered cannot exceed the vehicle's registered seating capacity.
[119] 
[120] 6.3 Cancellation Policy
[121] A Passenger may cancel an Accepted or Pending ride request at any time.
[122] A Pool Host may cancel a scheduled ride at any time, but cancellation is blocked within 15 minutes of the stated departure time.
[123] An Admin may cancel any ride at any time, without the 15-minute restriction.
[124] When a Pool Host cancels a ride, all passengers who have Accepted requests on that ride are automatically notified.
[125] 
[126] 6.4 Account Suspension Policy
[127] An Admin may suspend any user account at any time.
[128] A suspended user is prevented from logging in and receives an "Account Suspended" error upon login attempt.
[129] Suspension is reversible; an Admin may re-activate a suspended account at any time.
[130] Grounds for suspension include: repeated cancellations, safety-related rating tags, verified complaints, or policy violations.
[131] 
[132] 6.5 SLA (Service Level Agreement) for Incidents
[133] All SOS/Incident tickets carry an automatic SLA deadline of 15 minutes from the time of creation.
[134] An Admin is expected to acknowledge and begin resolving an incident within this SLA window.
[135] Unresolved incidents past their SLA are visually flagged in the Admin Incident Queue.
[136] 
[137] 6.6 Rating & Dispute Policy
[138] Ratings may only be submitted after a ride has been marked as Completed.
[139] Ratings containing safety-critical tags (Harassment, Unsafe Driving, Rash Driving) are auto-escalated to the Incident Queue.
[140] Users who receive consistent negative safety ratings may be reviewed and suspended by an Admin.
[141] Rating content is stored permanently and may be used in dispute resolution.
[142] 
[143] 7. Security Architecture
[144] 7.1 Authentication & Session Management
[145] The platform uses JSON Web Tokens (JWT) for stateless, secure authentication.
[146] All passwords are hashed using bcrypt (10 salt rounds) before storage. Plaintext passwords are never stored.
[147] JWTs are signed with a strong, application-specific secret (JWT_SECRET) stored in a server-side .env file.
[148] The JWT_SECRET is mandatory; the server will refuse to start if it is absent.
[149] Tokens are set with a 15-minute expiry to minimise the impact of token theft.
[150] Sliding Session Management: The backend auto-renews tokens on each authenticated request by emitting an X-New-Token response header. The frontend intercepts this header and updates the stored token seamlessly, ensuring active users remain logged in without re-authentication.
[151] On receiving a 401 Unauthorized response, the frontend automatically clears the session and redirects the user to the login page.
